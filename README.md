## 📑 Задание 1. Статистика успеваемости

### 🎯 Описание задачи
На основе датасета `edu_lrs` требовалось выполнить следующие шаги:
1. Рассчитать процент выполнения заданий учениками.
2. Определить зоны успеваемости.
3. Построить сводные отчеты по трем срезам:
   * По преподавателю
   * По организации
   * По региону

### 🛠️ Реализация и инструменты
Были построены детальные агрегированные отчеты. Для реализации сложной логики в **Google Sheets** использовались такие формулы как:
* `ARRAYFORMULA`, `FILTER`, `BYROW`, `LAMBDA`
* `COUNTA`, `COUNTIF`, `SUMPRODUCT`

### 🧮 Примеры использованных формул

#### 1. Подсчёт количества учеников по конкретному преподавателю
```excel
=COUNTIF(edu_lrs!B$2:B$1000; B4)
```

#### 2. Расчёт учеников, выполняющих план на 80% и выше
```excel
=SUM(BYROW(FILTER(edu_lrs!F$2:P$1000; edu_lrs!B$2:B$1000=B4); LAMBDA(row; --(COUNTA(row)>=INT(COUNTA(edu_lrs!F$1:P$1)*0,8)))))
```
**Разбор логики работы:**
* `FILTER` — отбирает учеников конкретного преподавателя.
* `BYROW` — поочередно применяет расчет к каждой строке массива.
* `COUNTA` — подсчитывает количество заполненных (выполненных) заданий в строке.
* `LAMBDA` — задает кастомную логику для проверки выполнения плана (>= 80%).
* `SUM` — агрегирует и возвращает итоговое количество учеников, успешно выполнивших условие.

#### 3. Подсчёт количества учеников по регионам
```excel
=SUMPRODUCT(--(LEFT(edu_lrs!D\$2:D\$1000;2)=VLOOKUP(AA4;'Регионы'!A:B;2;FALSE)))
```
**Разбор логики работы:**
* `LEFT` — извлекает первые два символа из ИНН школы для определения региона.
* `VLOOKUP` — ищет название региона в справочной таблице и возвращает его код.
* `LEFT(...)=VLOOKUP(...)` — выполняет логическую проверку на соответствие кодов.
* `SUMPRODUCT` — преобразует массив ИСТИНА/ЛОЖЬ в числовой вид (`--`) и суммирует совпадения, отдавая итоговое число учеников.

## 📊 Задание 2. Построение графиков

### 🎯 Описание задачи
Требовалось визуализировать динамику образовательного процесса и построить графики, отражающие изменение выполнения заданий относительно установленного плана в следующих разрезах:
* По школам
* По регионам
* Во временной динамике (таймлайн)

### 🗄️ Подготовка данных
Для обеспечения корректной работы аналитических инструментов и построения графиков исходные данные были предварительно преобразованы и сведены во вспомогательную таблицу в формате **long-table**.

Исходные данные содержали даты выполнения заданий в отдельных колонках (`c0–c10`). Для удобства визуализации и построения графиков они были преобразованы в плоскую структуру со следующими столбцами:
* `Регион`
* `Школа`
* `Дата занятия`
* `Номер недели`

Каждая строка в новой таблице соответствует одному зафиксированному факту выполнения задания. Трансформация выполнена с помощью формулы:

```excel
=ARRAYFORMULA(QUERY(SPLIT(FLATTEN(A2:A1000 & "#" & B2:B1000 & "#" & F2:P1000); "#"); "where Col3 is not null"; 0))
```

**Разбор логики трансформации:**
* `“#”` — технический разделитель для склеивания данных из разных столбцов.
* `FLATTEN` — «выпрямляет» двумерный диапазон дат `F2:P1000` в один длинный вертикальный столбец.
* `SPLIT` — разрезает склеенную строку обратно на отдельные колонки по разделителю `#`.
* `QUERY(...;"where Col3 is not null")` — фильтрует массив, удаляя строки с пустыми датами (где задания еще не выполнены).
* `ARRAYFORMULA` — обеспечивает динамическую обработку всего массива данных без необходимости протягивать формулу вниз.

### 📊 Построение аналитических агрегатов
На базе трансформированной плоской таблицы были подготовлены промежуточные агрегаты для графиков:
* Сводные таблицы успеваемости.
* Недельные агрегаты динамики.
* Региональная аналитика.
* Аналитика в разрезе образовательных организаций (школ).

## 📈 Логика расчета плана и факта (Бизнес-логика)

Для оценки текущего прогресса относительно контрольных точек была разработана модель **линейного прогресса обучения**. 

### 📅 1. Расчет планового выполнения
Так как учебный курс включает в себя **11 практических заданий**, распределенных на **9 учебных недель**, плановое выполнение рассчитывалось как равномерное еженедельное накопление прогресса:
* **1 неделя** → `11%` выполнения курса
* **2 неделя** → `22%` выполнения курса
* ...
* **9 неделя** → `100%` (полное завершение курса)

### 🚀 2. Расчет фактического выполнения
Фактические метрики рассчитывались на основе накопительного прогресса реально сданных работ относительно максимально возможного объема.
* **Формула общего возможного объема:**  
  $$\text{Общий объем} = \text{Количество учеников} \times \text{Количество заданий (11)}$$

## ⚙️ Задание 4. ETL-процесс из API в DWH

### 🎯 Описание задачи
В рамках автоматизации сбора данных требовалось разработать конвейер данных (ETL Pipeline) для решения следующих задач:
1. Получить исходные данные из внешнего REST API.
2. Преобразовать сложную вложенную JSON-структуру в плоский нормализованный вид.
3. Организовать надежную загрузку данных в аналитическое хранилище (PostgreSQL DWH).

### 📐 Архитектура решения
Перемещение и трансформация данных происходили по классической схеме:
$$\text{Внешний API} \longrightarrow \text{Python ETL Скрипт} \longrightarrow \text{PostgreSQL DWH}$$

---

### 🔄 Этапы пайплайна

#### 1️⃣ Извлечение данных (Extract)
Для выполнения HTTP-запросов к API использовалась библиотека `requests`. В коде реализована промышленная обработка исключений для отказоустойчивости скрипта:
* Контроль времени ожидания ответа (`timeout`).
* Обработка сетевых сбоев (`connection errors`), HTTP-ошибок ответов сервера и ошибок парсинга некорректного JSON.

#### 2️⃣ Преобразование данных (Transform)
Исходный JSON-ответ содержал глубоко вложенные массивы контактов (`PHONE` и `EMAIL`). Для соблюдения правил нормализации реляционных баз данных и устранения избыточности, единый объект был разделен на три независимые связанные сущности:
* `clients` — основные профили клиентов.
* `phones` — телефонные номера (связь «многие к одному» к клиенту).
* `emails` — адреса электронных почт (связь «многие к одному» к клиенту).

#### 3️⃣ Загрузка данных (Load)
Инструменты загрузки: библиотека `pandas` для формирования датафреймов и `sqlalchemy` в качестве ORM-движка. 
* Запись данных в СУБД выполнялась через оптимизированный метод `DataFrame.to_sql()`.

### 🗄️ Структура таблиц в DWH (Схема данных)

Ниже представлены SQL-скрипты развертывания целевых таблиц в хранилище:

#### Таблица клиентов (`clients`)
```sql
CREATE TABLE clients (
    client_id integer PRIMARY KEY,
    title text,
    first_name text,
    last_name text,
    status_id text,
    source_id text,
    client_type text,
    preferred_contact_method text,
    loaded_at timestamp DEFAULT now()
);
```

#### Таблица телефонов (`client_phones`)
```sql
CREATE TABLE client_phones (
    phone_id integer PRIMARY KEY,
    client_id integer REFERENCES clients(client_id),
    phone_number text,
    is_main boolean,
    loaded_at timestamp DEFAULT now()
);
```

#### Таблица электронных адресов (`client_emails`)
```sql
CREATE TABLE client_emails (
    email_id integer PRIMARY KEY,
    client_id integer REFERENCES clients(client_id),
    email_type text,
    email_address text,
    email_category text
);
```

## 🌟 Задание со звёздочкой. Apache Airflow DAG

Для перевода разработанного скрипта в промышленную эксплуатацию и регулярного обновления данных в DWH был спроектирован и запущен регулярный граф задач — **Apache Airflow DAG**.

### 🔄 Реализованный Pipeline автоматизации
Автоматический запуск по расписанию оркеструет следующие шаги:
$$\text{Извлечение из API} \longrightarrow \text{Валидация и трансформация} \longrightarrow \text{Загрузка дельт в PostgreSQL}$$

### 🛠️ Использованные компоненты Airflow
* `DAG` — декларативное описание графа и логики его выполнения.
* `PythonOperator` — для исполнения логики ETL-модулей, написанных на Python.
* `schedule` — настроен регулярный интервал запуска пайплайна.
* `retry logic` — автоматический перезапуск тасок при кратковременных сетевых сбоях API.
* `tags` — тегирование для удобной фильтрации в веб-интерфейсе Airflow.
* `catchup=False` — предотвращение лавинообразного запуска прошлых периодов при активации DAG.
